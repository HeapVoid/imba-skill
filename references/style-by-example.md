# Imba Style by Example

This is a side-by-side of two real components from the same project: a navigation/tabs UI. The first was generated by an LLM. The second was hand-written by an experienced Imba developer to replace it. **Same problem, same project, ~4× difference in length.** Read both, then read the lessons at the bottom.

## ❌ Verbose version (153 lines, generated)

```imba
import { app } from '../app.imba'

# ----------------------------------
# Navigation Bar (left strip, 80px)
# ----------------------------------
tag nav-bar
	css self
		w: 80px
		h: 100vh
		bg: #191919
		d: flex fld: column ai: center
		pos: relative
		fl: none
		mih: 0

		.top-section
			d: flex fld: column ai: center g: 8px
			margin-top: 16px

			.nav-icon
				w: 48px
				h: 48px
				rd: 16px
				bg: #1E1E1E
				d: flex ai: center jc: center
				cursor: pointer
				tween: opacity .2s
				pos: relative

				&.logo
					.logo-text
						fs: 22px
						fw: 700
						c: #444444

				&.peoples
					d: none
		# ... (130+ more lines of the same pattern)
```

## ✅ Concise version (41 lines, human-written)

```imba
tag nav-tabs
	activeTab = 0
	change

	def changeTab tab
		activeTab = tab
		change!

	css self
		p: 4px rd: 8px
		bg: $base9
		bxs: inset 0 0 0 1px black/10
		box-sizing: border-box

		.tabs
			pos: relative
			d: flex ai: center

		.active-bg
			pos: abs t: 0 b: 0
			rd: 4px
			bg: $blue6
			bd: solid 1px $blue5
			tween: cubic-in-out .2s
			&.right l: calc(50% + 2px)

		.tab
			pos: rel
			cursor: pointer
			d: flex ai: center jc: center
			fs: 14px lh: 24px fw: 400 c: $base6 tt: capitalize
			w: 50% rd: 6px p: 4px
			tween: ease-in-out .2s
			&:hover c: $base3
			&.active c: $blue1

	<self>
		<.tabs>
			<.active-bg [l: {100 / data.length * activeTab}% w: {100 / data.length}%]>
			for tab, idx in data
				<.tab .active=(activeTab == idx) @click=changeTab(idx)> tab
```

## What makes the second one shorter

| Lever | Verbose | Concise |
|---|---|---|
| **Multiple props per CSS line** | one prop per line | `pos:rel cursor:pointer d:flex ai:center jc:center` |
| **Theme tokens** | `#191919`, `#1E1E1E`, `#444444` | `$base9`, `$blue6`, `$base3`, `$blue1` |
| **Color/opacity syntax** | would write `rgba(0,0,0,0.1)` | `black/10` |
| **`tween:` for transitions** | `transition: all .2s …` | `tween: cubic-in-out .2s` |
| **Inline computed positioning** | mint a CSS class for each position | `<.active-bg [l:{100/n*i}% w:{100/n}%]>` |
| **Binary state, not mirrored** | both `.active=…` and `.inactive=…` | only `.active=…`; default state is inactive |
| **Flat structure** | nested `top-section`/`spacer`/`center-section`/`spacer`/`bottom-section` | flat children with `g:` for gaps |
| **No decorative comments** | `# ---- Section Title ----` blocks | none — structure speaks |
| **No dead UI** | `.peoples { d:none }`, `.support-btn { d:none }` placeholders | only what's actually used |
| **Callback prop as bare field** | `def open-profile do emit 'profile-open'` + parent listening | `change` declared as plain field, called `change!`, parent passes `<my-tag change=fn>` |
| **Shortest property names** | `margin-top: 16px`, `background:`, `cursor:` | `mt:16px`, `bg:`, `cur:` |

## Things to internalize

1. **CSS blocks are dense.** Group related properties on one line. The eye reads them as a unit. The long-form version is harder to scan, not easier.

2. **A project's design system exists for a reason.** Before writing any color, find the project's `css.imba` (or equivalent) and use the tokens defined there. Hex values mark code as imported from outside.

3. **Inline `[…]` styles for one-off computed values** beat creating CSS classes you'll only use once. Especially for positioning derived from props or state.

4. **The default state is the unstyled state.** If something is "active when X, inactive when not X", style only the active branch. Style for the negative case as well, and you've doubled your CSS.

5. **Flat beats nested.** Wrapper divs to group "the top stuff" and "the bottom stuff" rarely earn their cost. Use flexbox `g:` for gaps and let the parent flex container do the work.

6. **Don't ship hidden UI.** A `.support-btn { d:none }` rule is a sign you added an element you didn't need. Delete it.

7. **Callback props are just fields.** Imba's idiom for "this component fires events upward" is to declare a field with no value (`change`) and call it as `change!`. The parent assigns: `<nav-tabs change=on-tab-change>`. No `emit`, no event objects, no subscription.

8. **Decorative comments are noise.** Imba is structural enough that section headers add nothing. The `tag` definition, the `css self` block, the template — these are the sections.

When you write or review Imba, ask: *can each CSS rule be one line shorter? can each block be denser? is every property using its shortcut? are colors coming from the theme?* The answer is almost always yes.

# Part 2 — More patterns from a real refactor

The same project's "Design part 1" commit replaced ~669 lines of generated code with ~433 hand-written lines across 13 files (chats-list, messages-list, project-bar, css.imba, etc.). Below are the patterns the human used that the LLM had missed. Each one shows up dozens of times across the diff.

## Layout

- **Use CSS Grid for app shells, not flex+overflow hacks.** The generated code did `d:flex fld:column` + `fl:1` + `mih:0` + `ofy:auto` + `pos:absolute bottom:16px` to pin a footer. The human rewrite is one line: `d: grid gtr: auto 1fr auto g: 16px`. Header / scrollable middle / footer just *are* the three rows. No absolute positioning, no overflow gymnastics.
- **`pos: rel` / `pos: abs`**, never `position: relative`.
- **`s: 48px`** = `w:48px h:48px`. **`s: 100%`**, **`s: 50% rd: full`** for round avatars.
- **`d: flex ja: center`** = `ai:center jc:center` (justify+align combined). Use this everywhere you'd write both.
- **`rd: full`** for fully rounded.
- **`aspect-ratio: 1`** for squares whose size depends on the parent — better than fixed `w:Xpx h:Xpx`.
- **`bxs: inset 0 0 0 1px black/10`** for borders that don't shift layout (no border, just an inset shadow).
- **Compact positioning:** `inset: 0 r: unset my: auto l: -24px` instead of four separate `t/r/b/l` lines.

## Pseudo-states and parent selectors

- **`bg@hover: $base8`** — apply a value only on hover, in one line. Same for any property: `c@hover: $font0`. Beats writing `&:hover bg: $base8` on its own block when there's only one property.
- **`c^@hover: $base6`**, **`c^.active: $blue5`** — caret-prefix means "respond to the *parent's* state". Lets a child react to the parent's `:hover` or `.active` without nesting selectors.
- **Combine the two**: `c: $base7 c^@hover: $base6 c^.active: $blue5` — three states on one line.
- **Caret count = levels up.** `^` = parent (1 up), `^^` = grandparent, `^^^` = great-grandparent. **`..`** = ANY ancestor up to root (use it for theme hooks like `c..@dark: warm1`).

## Color tokens with opacity — only on built-in literals

- **`black/10`**, **`white/20`**, **`blue6/40`**, **`warm8/50`** — slash-syntax for translucency works on **built-in palette names** (which the compiler inlines as `hsla()` literals and rewrites the alpha at compile time).
- **It does NOT work on `$`-prefixed CSS-variable tokens** you define yourself: `$base8/50` ❌ — `$base8` becomes `var(--base8)` at runtime, and CSS variables don't support a slash-alpha syntax.
- **Correct pattern:** define a *new* token at the desired opacity using the literal once, then use the new token plain.

```imba
global css html
	$base8: warm8
	$base85: warm8/50    # ✅ alpha on the literal — this works
	$brand: blue6
	$brand-dim: blue6/30 # ✅

css .x
	bg: $base85          # ✅ no slash on the token
	bg: $base8/50        # ❌ broken — runtime var can't be slashed
	bg: warm8/50         # ✅ literal directly — works for one-offs
```

## Theme tokens defined globally

`css.imba` should look like this:

```imba
global css html
	$base1: warm1
	$base2: warm2
	$base85: warm8/50
	$blue5: blue5
	$blue6: blue6

	body
		ff: Inter, sans-serif
		c: $base3
		bg: $base9
		m: 0

	h1, h2, h3, h4, h5, h6, p m: 0
	h2 fs: 20px lh: 24px fw: 400
	p fs: 14px lh: 20px fw: 400
		&.light c: $base6

	button
		cursor: pointer
		ol: none us: none
		d: flex ja: center g: 8px
		fs: 14px lh: 20px
		c: $base3 p: 13px 20px rd: 8px
		bd: solid 1px black/10
		tween: ease-in-out .2s
		&.dark
			c: $base6 c@hover: $base4
			bg: $base9
```

Two things to copy from this:

1. **Tokens are remapped from Imba's built-in palette names** (`warm1`…`warm9`, `blue1`…`blue6`) to project-semantic names (`$base1`…`$base9`). The built-in names *are* the source — use them, don't reinvent with hex.
2. **Base typography and button styles live globally**, with variants as classes (`p.light`, `button.dark`). Components then just write `<h2>`, `<p .light>`, `<button .dark>` and inherit everything. Don't redefine font-size on every `<span.title>` you create.

## Use semantic HTML, not divs with classes

The verbose code wrote `<span.project-title>` with custom CSS for fs/fw/c. The human rewrite is `<h2> project-name`. Same for `<p .light> description` instead of `<span.project-desc>`. The browser already has tags for these — use them, and the global CSS handles styling.

## One-line tags

When a tag has a single child, write them on one line:

```imba
<self> <.spinner>
<.icon-box> <user-icon>
<.empty> <simple-loader [s: 48px]>
<.progress> <.bar [w: {(done * 100) / total}%]>
```

Don't break it into two indented lines unless the child has its own children worth indenting.

## Inline computed inline-style for derived width

`<.bar [w: {(completed * 100) / total}%]>` — progress bar without a single line of "fill" CSS. The parent `.progress` defines the track; the child's width is computed inline from state. No CSS class per width.

## Animation shorthand

```imba
@keyframes spin
	to transform: rotate(360deg)
```

Use `to` (not `100%`) and skip the `0%` start frame entirely. `tween: ease-in-out .2s` for transitions; `animation: spin 1.5s cubic-in-out infinite` for keyframe loops.

## Safe navigation in templates

`app.projects.active..id == project.id` — the `..` operator returns `undefined` if the LHS is null/undefined, no need for an `if` guard. Use it freely in conditional class bindings.

## Reusable icon system via tag inheritance

Instead of inlining `<svg>` in every component or importing icon files, define:

```imba
tag custom-icon
	paths = []
	size = {w: 16, h: 16}
	css self
		$icon-color: $base6
		d: flex ja: center
		svg tween: ease-in-out .2s
	<self>
		<svg width="{size.w}px" height="{size.h}px" viewBox="0 0 {size.w} {size.h}">
			for path in paths
				<path d=path>

tag stroke-icon < custom-icon
	css self
		svg
			stroke: $icon-color
			stroke-width: 2px
			stroke-linecap: round
			stroke-linejoin: round

tag plus-icon < stroke-icon
	paths = ["M8,1 L8,15 M1,8 L15,8"]
```

Then in any component: `<plus-icon [$icon-color: blue5]>`. Color is a CSS variable so callers override it inline. New icons are 2 lines (just `paths = [...]`).

## Things the human deleted that the LLM had added

- **Decorative section comment blocks** (`# ----- Header -----`).
- **Reset CSS in global.css** (`* { margin:0; padding:0 }`, `*:focus { outline:none }`, `img,picture,video {...}`). Imba's CSS layer + sensible component-level resets cover this; ~60 lines of generated reset got dropped.
- **Element-specific CSS for things that should be global** (`body { font-family: ... }` was in global.css; the rewrite moved it into `css.imba` next to the tokens).
- **Nested wrapper divs** (`.right-col > .content-row > .chat-item-wrap > .chat-item`) — collapsed to a single grid container with direct children.
- **Mirrored state classes** — `.active=… .inactive=…` became just `.active=…`.
- **Form state that isn't used yet** — the verbose chats-list had a whole `creating` / `new-chat-name` / `start-create` / `confirm-create` / `cancel-create` flow inline. The rewrite commented it all out and replaced it with a single `def create-room` that just does `app.chats.create('New room')`. **Don't build features that aren't asked for.**
- **`fl: 1` + `mih: 0` + `ofy: auto`** chains for "make this fill the parent and scroll" — replaced by grid `1fr` row.

## The distilled rule

> Every line of CSS, every wrapper div, every helper method, every comment — make it justify its existence. If a grid row, a global token, a semantic HTML tag, or a one-line shortcut can replace it, it should.
